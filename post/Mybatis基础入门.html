Mybatis基础入门

使用Mybatis连接数据库。Mybatis本身其实就是对JDBC的封装，使得Mybatis使用起来比JDBC还要简单。

1. 第一个Mybatis程序

在开发第一个Mybatis程序之前，先确定使用的数据库，目前使用的数据库为MySQL数据库。

1.1 jar依赖

需要下载Mybatis的依赖jar，可以使用maven来创建Java项目，这样就可以自动把项目所依赖的jar加载到项目中，Mybatis项目主要使用两个jar包分别是：

  • Mybatis的核心jar包
  • 数据库驱动jar包：mysql-connection-java

对应的maven依赖写法如下：

<dependencies>
    <!-- mybatis的依赖-->
    <dependency>
      <groupId>org.mybatis</groupId>
      <artifactId>mybatis</artifactId>
      <version>3.5.7</version>
    </dependency>

    <!-- mysql驱动的依赖 -->
    <dependency>
      <groupId>mysql</groupId>
      <artifactId>mysql-connector-java</artifactId>
      <version>8.0.31</version>
    </dependency>

</dependencies>

1.4 XML配置文件

现在实体类已经有了，数据表也有了，现在需要做的就是把Java中的实体类和数据库中的user表联系起来，那么使用的工具就是Mybatis。那么就需要写两个配置文件：

  • mybatis.xml，作用是：Mybatis的核心配置文件，需要填写链接的数据类型、用户名、密码等信息。
  • UserMapper.xml，作用是做实际的SQL操作。一个XXX数据表对应一个XXXMapper.xml文件。

mybatis.xml的配置如下：

<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE configuration
        PUBLIC "-//mybatis.org//DTD Config 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-config.dtd">
<configuration>
    <environments default="development">
        <environment id="development">
            <!-- JDBC – 这个配置直接使用JDBC 的提交和回滚功能。它依赖于从数据源获得连接来管理
                   事务的生命周期。
               • MANAGED – 这个配置基本上什么都不做。它从不提交或者回滚一个连接的事务。而是让
                   容器（例如：Spring 或者J2EE 应用服务器）来管理事务的生命周期
            -->
            <transactionManager type="JDBC" />
            <!--
           数据源类型：
               UNPOOLED – 这个类型的数据源实现只是在每次需要的时候简单地打开和关闭连接。
               POOLED – 这个数据源的实现缓存了JDBC 连接对象，用于避免每次创建新的数据库连接时都初始
               化和进行认证，加快程序响应。并发WEB 应用通常通过这种做法来获得快速响应。
               JNDI – 这个数据源的配置是为了准备与像Spring 或应用服务器能够在外部或者内部配置数据
               源的容器一起使用，然后在JNDI 上下文中引用它
        -->
            <dataSource type="POOLED">
                <property name="driver" value="com.mysql.cj.jdbc.Driver" />
                <property name="url" value="jdbc:mysql://localhost:3306/test?useUnicode=true&useSSL=false&characterEncoding=UTF8&serverTimezone=UTC" />
                <property name="username" value="root" />
                <property name="password" value="cukor" />
            </dataSource>
        </environment>
    </environments>
    <mappers>
        <mapper resource="UserMapper.xml"/>
    </mappers>
</configuration>

UserMapper.xml的配置如下：

<?xml version="1.0" encoding="utf-8" ?>
<!DOCTYPE mapper
        PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<!--
	mapper：根标签
	namespace：命名空间
-->
<mapper namespace="com.tipdm.UserMapper">
    <select id="queryById2" resultType="com.tipdm.User">
        select *
        from user
        where id = #{id}
        <!-- #{XX}表示占位符的意思 -->
    </select>
</mapper>

这个东西是maven项目的配置文件，作用就是自动把Java需要的jar导入到当前项目中来，不需要程序员手动配置。关于pom.xml的写法也不需要程序员记住，因为太繁杂，所以有一个网址直接把maven需要依赖的jar文件的写法已经写好了，这个就是maven中心仓库，使用到一个新的jar包需要知道怎么加载它的时候直接到这个中心仓库的网址中查找即可。

中心仓库地址：Maven Repository: Search/Browse/Explore

3. Mybatis的一般项目目录

  • com.tipdm：项目总包名

  • domain：对应数据库中数据表的Java源文件。

  • mapper（dao）：对应Java数据源和数据库表的映射接口

  • test：用于测试的包

  • resources：资源文件，在资源文件中的文件Java源文件默认访问的时候就是类的根目录。

    • mapper：一个数据表对应一个XXXMapper.xml文件，内容是实际的SQL语句

SqlSessionUtil.java：对SqlSession的获取代码进一步封装，让代码更加简洁。

package com.tipdm.utils;

import org.apache.ibatis.io.Resources;
import org.apache.ibatis.session.SqlSession;
import org.apache.ibatis.session.SqlSessionFactory;
import org.apache.ibatis.session.SqlSessionFactoryBuilder;

import java.io.IOException;
import java.io.InputStream;

public class SqlSessionUtil {
    private SqlSessionFactory sqlSessionFactory = null;

    public SqlSessionUtil(String config) {
        // SqlSessionFactoryBuilder
        SqlSessionFactoryBuilder sqlSessionFactoryBuilder = new SqlSessionFactoryBuilder();
        // SqlSessionFactory
        InputStream resourceAsStream = null;
        try {
            resourceAsStream = Resources.getResourceAsStream(config);
        } catch (IOException e) {
            e.printStackTrace();
        }
        sqlSessionFactory = sqlSessionFactoryBuilder.build(resourceAsStream);
    }

    public SqlSession getSqlSession() {
        // SqlSession
        return sqlSessionFactory.openSession();
    }
}

4.3 接口代理方式

上面是对原来第一个Mybatis程序的进一步封装，下面采用接口代理的方式重新写一下使用Mybatis做查询的操作。

所谓接口代理方式，就是在把SqlSession封装成一个类之后的再次升级，使用接口来管理的话，代码量就减少了很多。

其他地方不需要做改变，只是在测试类中调用的时候，使用的是SqlSession的对象调用getMapper方法的方式。

例如：

/**
 * 每个测试执行前都会执行@Before注解的方法
 */  
@Before
public void init() {
    sqlSession = new SqlSessionUtil("mybatis.xml").getSqlSession();
}

/**
 * 每个测试执行后都会执行@After注解的方法
 */
@After
public void after() {
    sqlSession.close();
}   
@Test
public void test1() {
    //获取UserMapper接口的具体代理对象
    UserMapper mapper = sqlSession.getMapper(UserMapper.class);
    User users = mapper.selectById(3);
    System.out.println("users = " + users);
}

上面的代码直接通过mapper调用selectById方法，这个是接口中的方法声明，具体的实现会映射到UserMapper.xml中的id为selectById的标签。

<select id="selectById" resultType="com.tipdm.domain.User">
    select *
    from test.user
    where id = #{id};
</select>

执行之后的结果如下：

4.4 添加

既然已经使用了接口代理的方式，那从这往下之后就直接使用接口代理的方式来写代码了。

在UserMapper接口中增加下面的代码：

/**
 * 插入一条记录
 */
int insertUser(User user);

然后在对应的UserMapper.xml中添加下面的代码：

<insert id="insertUser">
    insert into test.user (name, age, sex, address, birth)
    values (#{name}, #{age}, #{sex}, #{address}, #{birth})
</insert>

因为主键id是自动增加的，所以做插入的时候可以不做指定。

然后在测试类中添加下面的代码：@Before和@After同上，下面就不再赘述了。

@Test
public void test4() {

    Calendar calendar = Calendar.getInstance();
    Date time = calendar.getTime();
    User user = new User("Jay", 44, '男', "台湾", time);

    System.out.println("i = " + i);

i = 1

这是因为我们做了数据的增加，所以user表有一行做了改变，所以返回1给i。

但是，我们通过数据SQL语句来对user表做下面的查询的时候，并没有发现我们刚刚增加的记录。

select * from user;

这是因为没有提交导致的，之前做查询的时候不需要做提交，是因为查询不会对数据表进行修改，所以也就没有提交的必要，但是插入、删除和更新都会导致数据表的修改，所以在做这些操作的时候记得做提交操作。

提交的代码也不麻烦只需要在测试的代码下面添加下面的代码即可。

sqlSession.commit();

添加了这行代码之后再去运行程序，然后再对数据表user做一个查询操作，即可得到下面的结果：

说明“Jay”这条记录就真正被插入到user表中了，并且它的id为9，而在插入”Jay”这条记录之前的记录是”David”他的id是7，这是因为我们上一次没有真正提交，导致8这个id被占用了。

然后运行之后，终端输出：

i = 1

然后通过数据库查表，得到下面的结果：

可以看到id为4并且姓名为“赵六”的记录已经被成功删除了。因为我们这次直接做了提交，所以它就会直接改变。

6. Mybatis映射配置文件详解

6.1 结果映射

在mybatis中，默认将查询的结果返回一个实体类对象，查询结果和实体类根据结果列别名和类中属性名一一对应。当查询结果列别名和类属性名不一致。解决方案有两种：

  • 将别名修改为一致
  • 使用mybatis内置结果映射处理器

例如：

实现类：

public class Student {
    private Integer id;
    private String stuName;
    private Integer stuAge;
    private String address;
}

而对应的数据表：

create table student (
	id int primariy key,
    stu_name varchar(255),
    stu_age int,
    address varchar(255)
)

此时stuName、stu_name和stuAge和stu_age不对应。那么查询出来的结果就不对应。可以通过别名的方式来做修改。例如：

select id, stu_name stuName, stu_age stuAge, address
from student;

这样就可以查询出正确的结果。但这种方式对于字段比较多的表就不太适合了。

所以当遇到字段比较多的数据表做查询操作的使用应当使用mybatis内置结果映射处理器方式来做查询比较好。

6.4 #和$(重点)

在mybatis中，会将开发者定义的sql进行解析，解析分为以下2类sql：

  • 静态sql ，在解析时，直接将参数拼接到sql中，这种就是静态sql conn.createStatement()

  • 动态sql，在解析时，会使用?这个占位符，替代参数 conn.prepareStatement()

这两种解析方式，mybatis是根据${}和#{}进行区分的。

${}的sql是静态sql ${uName}

#{}的sql是动态sql #{uName} #{arg0} #{param1} #{0}

不论是静态sql，还是动态sql都能获取传递参数，但是${}是使用的字符拼接，#{}使用PreparedStatement进行参数的预处理。

在一定程度上说，${}能实现的功能，#{}都能实现，并且由于#{}PreparedStatement进行SQL的预处理，一定程度上可以防止SQL注入攻击。所以在开发中，能使用#{}尽量使用#{}。
PreparedStatement预处理的本质是将参数进行转换为字符串，当做参数字符串处理。所以，如果参数信息是一个特殊的关键字，例如： 数据库名，表名，函数名，内置关键字，使用预处理，则关键字转为了字符串，无效了，此时必须使用字符串拼接。

  • mybatis中，提供了一个RowBounds这个类，用于进行分页查询数据
  • 使用分页插件:PageHelper(推荐）

10.1 方案一[rowBounds]

使用RowBounds这种分页查询方式，在实际开发中是远远不够使用的，这种方式只是返回的相应的数据，总数据条数，页数，分页起始页都需要开发者自己处理。

例子：

在StudentMapper.java接口中添加下面的代码：

/**
 * mybatis 推荐的分页方式
 * @param rowBounds
 * @return
 */
List<Student> selectPage(RowBounds rowBounds);

在StudentMapper.xml中添加下面的代码：

<select id="selectPage" resultMap="BaseResultMap">
    select
    <include refid="base_column"/>
    from test.student
</select>

在测试类中添加下面的代码：

@Test
public void test11() {

10.2 方案二[分页插件]

市面主流分页插件: PageHelper

  1. 导入PageHelper的相关jar包

在pom.xml文件中添加下面的内容：

        分页插件 ：PageHelper

在PageHelper，每次开启分页，只对最近的一次查询有效。如果要多次分页查询，需要多次开启。

PageHelper是利用，拦截器原理，拦截最近的一次查询，然后在其基础拼接分页信息。PageHelper只对最近一次查询有效，这种模式，对本身的程序没有侵入，所以，本身程序该怎么写还是怎么写。

11. 新增自增长(重点)

在mybatis中，支持数据库的自增长功能，因为在某些特殊的业务场景中，当前数据的ID，可能是另外某些数据的业务ID。

Mybatis在insert指令中，提供了3个属性：useGeneratedKeys、keyProperty、keyColumn

  • useGeneratedKeys：表示使用数据库自增长

  • keyProperty：自增长的列对应的类中的属性

  • keyColumn：自增长的列

Mybatis会自动将增长的值，封装到传入参数的属性中。

例子：

在StudentMapper.java接口中添加下面的代码：

/**
 * 插入一条记录，让它的id自增长
 */
int insertStu(Student student);

在StudentMapper.xml文件中添加下面的代码：

<insert id="insertStu" parameterType="Student" useGeneratedKeys="true" keyProperty="id" keyColumn="id">
    insert into test.student (stu_name, stu_age, address)
    values (#{stuName} ,#{stuAge}, #{address})
</insert>

需要特别注意的是：SQL语句的values后面的这些引用变量应当和实体类中的属性保持一致。

然后在测试类中添加下面的代码：

12. 动态SQL(重点)

动态SQL是指根据不同的参数，产生不同的SQL语句。这样的SQL就是动态SQL。

Mybatis提供了一些标签，用于动态SQL的生成:

  • if
  • foreach
  • choose
  • where: 可以去除前置多余的and、or等挂关键字;不能去除后置
  • set
  • trim

12.1 if标签

MyBatis if 类似于 Java 中的 if 语句，是 MyBatis 中最常用的判断语句。使用 if 标签可以节省许多拼接 SQL 的工作，把精力集中在 XML 的维护上。

在if标签中，test属性是必须有，test属性值是一个表达式，如果表达式值为true,则if标签包裹的内容会拼接在当前sql上。

12.2 foreach

foreach标签可以帮助你根据集合或数组中的元素动态地生成SQL语句。它可以让你避免手动拼接SQL语句，从而提高开发效率。

foreach标签用于循环遍历集合或数组，并为每个元素执行SQL语句。它的语法格式如下：

<foreach collection="集合或数组" item="循环变量" index="索引变量" open="开始符" close="结束符" separator="分隔符">
    SQL语句
</foreach>
  • collection: 指定要循环的集合或数组。
  • item: 指定循环变量的名称。
  • index: 指定索引变量的名称。在遍历List和数组时，index是元素的序号；在遍历Map时，index是元素的key，item是Map的值。
  • open: 指定循环开始时要添加的字符串。
  • close: 指定循环结束时要添加的字符串。
  • separator: 指定分隔符，用于分隔每次循环生成的SQL语句。

下面是一个使用foreach标签进行批量插入数据的例子：

在StudentMapper.java中添加下面的代码：

/**
 * 使用foreach循环插入数据

int insertList(List<Student> studentList);

在StudentMapper.xml中添加下面的代码：

<insert id="insertList">
    insert into test.student (stu_name,stu_age, address)
    values
    <foreach collection="list" item="item" separator=",">
        (#{item.stuName}, #{item.stuAge}, #{item.address})
    </foreach>
</insert>

上面代码等效：

12.3 choose

choose标签用于在多个条件中选择一个条件来执行。它类似于Java中的switch语句。choose标签通常与when和otherwise标签一起使用。下面是一个简单的例子：

在student表中，按下面的条件做查询：

  1. 当年龄大于30的时候就按地址来查
  2. 当年龄大于40的时候就按姓名来查
  3. 否则就按id来查。

在StudentMapper.java接口中添加下面的代码：

/**
 * 当年龄大于30的时候就按地址来查，当年龄大于40的时候就按姓名来查，否则就按id来查。
 */
List<Student> selectByCondition(Student student);

在StudentMapper.xml中添加下面的代码：

<select id="selectByCondition" resultMap="BaseResultMap" parameterType="Student">
    select <include refid="base_column"/>
    from test.student
    where
    <choose>
        <!-- 条件一 -->
        <when test="stuAge > 40">
            stu_name = #{stuName}
        </when>
        <!-- 条件二：当条件一不满足的情况下才会进入条件二的判断 -->
        <when test="stuAge > 30">
            address = #{address}
        </when>
        <!-- 当上面的条件都不满足了才会进入到最后的这个 -->
        <otherwise>
            id = #{id}
        </otherwise>
    </choose>
</select>

在测试文件中添加下面的代码：

    student.setId(8);
    List<Student> students = mapper.selectByCondition(student);
    for (Student student1 : students) {
        System.out.println("student1 = " + student1);
    }
    sqlSession.close();
}

  • mybatis 提供了where标签，取代where关键字，默认去掉第一个条件 and

注意：建议，只在查询语句中使用where标签，因为当where标签中的条件都不成立时，会没有where关键字。

13. 缓存

mybatis为了提高查询效率、性能。提供了缓存，缓存分为2类:

  • 一级缓存

  • 二级缓存

一级缓存：是指SqlSession级别的缓存，在同一个SqlSession，同样的SQL语句只会执行一次，不是第一次执行的SQL会从缓存中获取数据。

二级缓存: 是指SqlSessionFactory级别的缓存，在同一个SqlSessionFactory中，同样的SQL语句，只会执行一次，不是第一次执行的SQL会从缓存中获取数据。

注意：不论一级缓存还是二级缓存，否是JVM中缓存。当服务器有多台时，缓存容易发生无效，数据发生了更新。缓存没有更新。在实际使用，如果数据可能发生比较频繁的更新不建议使用mybatis。

16. 联表查询(难点)

16.1 多表查询解决方案

  1. 写实体类，通过定义实体类，将多表的所有字段都涵盖，解决多表查询问题，但是，这种方案只能解决多对一，但是无法解决一对多问题。

例如：写实体类可以解决，一个学生类中班级信息问题，但是无法解决班级类中学生信息问题。

学生信息是一个数组。

  1. 多次查询。将需要结果拆分成多个简单的单表查询，连续查询多次。使用Java代码处理查询结果。

例如：学生班级信息。

查询某个学生学生信息及其班级信息。

根据ID查询学生信息

从学生信息中获取班级ID

根据班级ID查询班级信息

查询所有学生的信息及其班级信息

查询所有学生(100)(注意学生数据量)

查询所有班级(Map 使用ID 作为key 对象就是value)

循环学生，获取班级ID，从班级数据中，获取班级信息

注意：mybatis解决联表查询的方法就是基于以上方案。
